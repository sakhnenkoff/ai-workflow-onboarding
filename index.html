    2. AI suggestions — start typing a command and show Warp's inline suggestions
    3. Searchable history — press Ctrl+R and search for a previous command

    "When you're running Claude Code, having clean command blocks means you can visually track what Claude did versus what you did."
Wispr Flow
  • Install and set up voice-to-prompt
    Install via Homebrew: brew install --cask wispr-flow Or download directly from wisprflow.ai/downloads.

    After install, macOS will prompt for permissions. Grant both:
    1. System Settings > Privacy & Security > Microphone — enable Wispr Flow
    2. System Settings > Privacy & Security > Accessibility — enable Wispr Flow

    Set a global hotkey (default is Option+Space). Sign in with Google, Apple, or email.

    Demo: Open Claude Code, press the Wispr hotkey, speak your prompt, and watch it transcribe directly into the input field.

    "I use this constantly — reading code in one window and voice-prompting Claude in the terminal. No context switching, no typing long prompts."
Claude Code Basics
  • Install and authenticate
    Recommended install (native installer — no Node.js dependency, auto-updates): curl -fsSL https://claude.ai/install.sh | bash Or via Homebrew: brew install --cask claude-code Then launch: claude First run will open a browser for authentication. Make sure they use their work Anthropic account.

    After auth, verify it works: claude --version
  • Bang command — execute bash instantly
    Type ! followed by any shell command directly in the Claude Code prompt. It runs instantly and adds the output to the conversation context — no back-and-forth with Claude needed. !git status !ls app-modules/Sources/ !cat AGENTS.md | head -20 "Instead of asking Claude to check git status and waiting for it to run the tool, just bang it. Instant, direct, and you control exactly what goes into context."
  • /btw — ask while Claude is working
    Use /btw to ask a quick side question while Claude is actively working. It answers without adding to the main conversation history — so it doesn't derail the current task. /btw what's the difference between Drive and FinnUI modules? "Claude is mid-task, reading files, making edits — and you have a quick question. /btw lets you ask without interrupting the flow."
  • Esc Esc — rewind to checkpoint
    Press Esc twice quickly to open the rewind menu. It shows checkpoints in the conversation — pick which point to rewind to. File changes get undone and the conversation is restored.

    You can also use the /rewind command for the same menu. "Think of it like Cmd+Z for AI — but you pick how far back to go. Claude went the wrong direction three steps ago? Rewind there, rephrase, and try again."
  • Shift+Tab — mode switching
    Cycles between three modes:
    • Normal — Claude asks permission before each action
    • Auto-Accept — Claude executes without asking (fast but less control)
    • Plan — Claude only plans, never executes (great for architecture discussions)

    "I use Plan mode first to let Claude think through the approach, then switch to Auto-Accept to execute. It's like having a senior dev plan and a junior dev type."
  • Ctrl+S — stash prompt
    Press Ctrl+S to stash your current prompt input. Think of it like git stash — saves what you're typing so you can do something else, then restore it later. "Midway through writing a complex prompt and need to do something else first? Ctrl+S stashes it so you don't lose your train of thought."
  • Session recovery — continue and resume
    claude --continue # Resume the last conversation claude --resume # Pick from any past session claude --resume my-feature # Resume a named session Name a session with /rename my-feature while inside Claude Code.

    "Terminal died? MacBook crashed? No problem. Your sessions are saved. --continue picks up right where you left off."
  • Thinking modes
    Add these keywords anywhere in your prompt to control reasoning effort. Claude uses adaptive reasoning — it dynamically allocates thinking based on the effort level:

    • think — standard reasoning effort
    • think hard — higher effort, more thorough
    • ultrathink — maximum effort for complex problems think hard about how to refactor this coordinator to support deep links "Use 'think' for straightforward stuff. 'ultrathink' when you need Claude to really reason through something — architecture decisions, tricky concurrency, multi-file refactors."
  • /context — visualize context usage
    Type /context to see a visual grid of current context usage. It shows optimization suggestions for context-heavy tools, memory bloat, and capacity warnings. "This tells you how much of Claude's context window you've used up. If things start feeling slow or off, /context shows you what's eating space."

CLAUDE.md / AGENTS.md Walkthrough
  • Open and walk through the context files
    Open the files together: cd ~/Developer/Finn/ios-app cat CLAUDE.md # Short — points to AGENTS.md cat AGENTS.md # The big one — walk through each section Key sections to highlight in AGENTS.md:
    • Available Skills table — the skill catalog
    • iOS Feature Pipeline — greenfield vs ported vs small task
    • Engineering Policies — SwiftLint, concurrency, design system rules
    • Build Verification — default simulator, when to ask before building
    • Git Policy — never commit unless explicitly asked
  • What context enables and how to shape it
    The key insight isn't "context makes Claude better" (obvious). It's that you control the context, and that's your superpower.

    Walk through three layers:

    1. CLAUDE.md / AGENTS.md — project-wide context
    This is always loaded. It tells Claude your architecture, policies, conventions, and available skills. Open it and show how specific it is — it's not generic boilerplate, it's our team's knowledge. # Show a specific policy that changes Claude's behavior: # e.g., the concurrency section that says "never assume SWIFT_DEFAULT_ACTOR_ISOLATION" # or the design system section that enforces Warp tokens over hardcoded values 2. Rules — behavioral overrides
    Rules in .claude/rules/ shape specific outputs. Show unleash-cli.md — it makes Claude format Unleash output as ASCII tables with all 8 environments. Without this rule, Claude would just dump raw JSON. "Rules are surgical. Instead of writing 'format this as a table' every time, the rule does it automatically. Think of them as linting for AI behavior." 3. Module AGENTS.md — scoped context
    Show .claude/rules/module-agents.md — when Claude touches files in a module that has its own AGENTS.md, it reads that module's specific conventions first. # Example: Recommerce/Sources/RecommerceTransaction/AGENTS.md # has TJT-specific patterns and API generation rules # Claude automatically picks these up when editing files in that module How to improve context:
    • Found Claude making the same mistake twice? Add a rule for it.
    • New architectural decision? Update AGENTS.md.
    • Module has unique patterns? Add a module-level AGENTS.md.
    • Repeating the same instructions? Create a skill.

    "Context isn't static documentation — it's a living system. Every time Claude gets something wrong, ask yourself: 'Could better context have prevented this?' If yes, update a file. You're training the system, not just using it."

AGENTS.md is basically a table of contents that keeps Claude grounded in reality."
Why This Matters (Philosophy)
  • Context is the multiplier
    "The model is the same for everyone. Context is what makes it yours. CLAUDE.md and AGENTS.md are the highest-leverage files in the entire repo — one set of files that makes every AI interaction better for every engineer on the team."
  • Skills encode team knowledge
    "Instead of 10 engineers each writing their own 'review my code' prompt — with varying quality — we write one skill that captures the best approach. A new engineer on day one gets the same review quality as a 5-year veteran."
  • When NOT to use AI
    Be honest about limitations:
    • Quick one-line fixes you already know how to do — just do them
    • Code you're deeply familiar with — your mental model is faster
    • Creative architectural decisions — AI is great at executing a vision, not great at having one
    • Debugging with runtime context — sometimes Instruments or LLDB is just better

    "AI isn't always faster. The real skill is knowing when to reach for it and when not to. If you catch yourself waiting 30 seconds for Claude to do something you could've done in 5, that's a signal to just do it yourself."
  • Token awareness
    Run /context in Claude Code to show the breakdown. Point out:
    • System prompt + AGENTS.md uses a chunk of tokens (always loaded)
    • Each MCP server adds context overhead
    • ultrathink uses ~32k thinking tokens — great for complex problems, overkill for simple ones

    "Think of tokens like a budget. You wouldn't use a sledgehammer to hang a picture. 'think' for simple tasks, 'ultrathink' for architecture. /context tells you where your budget is going."

Block 3: Orchestration
Claude Code as conductor, Codex as task manager
~50min ▶
Claude Code Power Features
  • Subagents — parallel work
    In Claude Code, ask it to do something that benefits from parallel research: Explore how the Jobs and Recommerce modules handle deep links — check both in parallel Claude will dispatch two subagents that work simultaneously, then merge results. You can also explicitly request parallel work: Use subagents to research these three ViewModels in parallel and summarize their patterns "Subagents are like junior devs you can send off to research things in parallel. Each gets its own context, works independently, and reports back with findings."
  • Hooks — automated behaviors
    Hooks run automatically in response to Claude Code events. They can be shell commands, HTTP calls, or prompts. Configured in settings files (~/.claude/settings.json for user-level, .claude/settings.json for project-level): # View/edit settings: claude /settings Example hook — run SwiftLint after every file edit: { "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "command": "swiftlint lint --path $FILE_PATH" }] } } Note: event names are PascalCase — PostToolUse, PreToolUse, etc. "Hooks are like CI, but local and instant. You can make Claude's environment enforce rules automatically — lint on save, format on write, whatever makes sense for your workflow."
  • Plan Mode — think before executing
    Press Shift+Tab to switch to Plan mode. In this mode, Claude only analyzes and plans — it never edits files or runs commands.

    Demo workflow:
    1. Switch to Plan mode (Shift+Tab until you see "Plan" in the status bar)
    2. Ask: "How would you add a new feature flag to the search screen?"
    3. Claude produces a detailed plan without touching any files
    4. Review the plan, adjust if needed
    5. Switch to Normal or Auto-Accept mode (Shift+Tab) and say "execute the plan"

    "Plan first, execute second. This prevents Claude from charging down the wrong path and making changes you'll have to undo."
  • Session management — named sessions
    /rename search-refactor # Name this session claude --resume search-refactor # Resume later from terminal "Name your sessions by feature or task. When you come back tomorrow, you resume with full context — Claude remembers everything from the session."

    "You can have 3 Codex tasks running in parallel, each on their own worktree, while you're doing something completely different in Claude Code. None of them interfere with each other."
How We Work With AI (Conceptual)
  • The two layers: Superpowers skills + repo skills
    There are two complementary layers:

    Superpowers skills (process layer) — general-purpose workflows that work on any project:
    brainstorming → writing-plans → subagent-DD → finishing
    Repo skills (domain layer) — iOS-specific skills that know our codebase:
    /ios-review-code /ios-pre-push /ios-write-unit-test /ios-write-snapshot-test /ios-pr
    You mix and match. For a big feature: brainstorm, plan, execute, review, pre-push, PR. For a small task: just use the repo skills directly.

    "The Superpowers pipeline is for when you want Claude to think through the problem before jumping in. The repo skills are for specific daily actions — reviewing, testing, pushing. You'll reach for the repo skills way more often."
  • When to use each approach
    Just use repo skills directly for:
    • Bug fixes, small changes, tests, code reviews
    • Anything you already know how to do — just want AI to execute faster

    Use the Superpowers pipeline for:
    • New features where you haven't decided the approach yet
    • Complex changes touching multiple modules
    • Porting from Android (use /ios-port-from-android to get the reference, then plan)

    "Most of your day-to-day will be repo skills — review, test, push. The full pipeline is for when you're building something new and want Claude to think before it acts."
  • Design-to-code with Figma MCP
    Design-to-code uses the Figma MCP server (set up in Block 4). The workflow is:
    1. Copy a Figma frame URL from the design file
    2. Paste it into Claude Code with a prompt like: Implement this Figma design as a SwiftUI view using Warp design tokens 3. Claude reads the Figma frame via MCP, gets a screenshot and layout info, and generates SwiftUI code using our Warp design tokens.

    "This closes the loop between design and code. No more eyeballing Figma specs — Claude reads the design and generates pixel-accurate SwiftUI."

Security Awareness (5min)
  • Ground rules for AI security
    Three rules:

    1. Never paste secrets into prompts. Instead of copying an API key into the chat, tell Claude to read the file: Read the Firebase config from GoogleService-Info.plist
    2. MCP tokens stay local. Auth tokens from Figma, Firebase, and Atlassian are stored in Claude Code's local config on the machine. Nothing is sent to Anthropic.

    3. Always review before committing. AI-generated code can have security issues — especially around auth flows, data handling, and network calls. The /ios-review-code skill explicitly checks for this.

Block 1: Warm-up — Guided Skill Runs
Build confidence with individual skills
~50min ▶
Quick recap before starting: open Warp, launch claude from the iOS repo root (cd ~/Developer/Finn/ios-app && claude). Skills are slash commands — type / in Claude Code to see the list.
Exercises — They Drive
  • Code Review
    Have them pick a recently changed file. To find candidates: !git log --oneline -10 Then invoke the review skill: /ios-review-code Claude will ask what to review. Point it at the file or a recent commit diff.

    Discussion points after output:
    • Did it catch Warp token usage vs hardcoded values?
    • Did it flag accessibility issues?
    • Did it check concurrency patterns?
    • Compare: "What would you have caught manually vs. what the skill found?"
  • Pre-push Validation
    Check out any branch with changes (or create a trivial one — e.g., add a comment to a file): /ios-pre-push This runs lint, unit tests, and snapshot validation. Show how it catches issues before CI does — saving 20+ minutes of CI feedback loop.

    "This is your local CI. Run it before pushing and you'll almost never get surprise CI failures."
  • Unit Test Generation
    Find a ViewModel or UseCase that has limited test coverage: !grep -rL "import Testing\|import XCTest" --include="*.swift" app-modules/Sources/ | head -5 Then invoke the test generation skill: /ios-write-unit-test Claude will ask which file to test. Point it at the ViewModel. Show how it generates tests using Swift Testing patterns (@Test, @Suite) matching the repo's conventions.

    "Notice how it uses Swift Testing, not XCTest — because AGENTS.md tells Claude our conventions. That's context doing the heavy lifting."
  • Snapshot Test
    Pick a SwiftUI view (e.g., something in app-modules/Sources/Drive/): /ios-write-snapshot-test Show how it generates both visual snapshot and accessibility snapshot tests, matching the project's test plan structure.

    "Snapshot tests are tedious to write by hand. This skill handles the boilerplate and gets the test plan structure right every time."
Optional Quick Wins
  • Warp Design System lookup
    Find a hardcoded color or spacing value in the code: /ios-warp-design-system What Warp token should I use instead of Color(hex: "#0063FB")? Shows how Warp tokens map to hardcoded values and enforces design consistency.
  • Localization flow
    Invoke the localization skill to add or update localized strings: /ios-localization-flow This handles the full workflow: adding keys to the strings file, running SwiftGen to regenerate accessors, and updating the SwiftUI views to use the generated constants.
  • iOS 26 compatibility check
    Run this skill to validate Liquid Glass gating in your code: /ios-ios26-compatibility-check It checks that iOS 26-specific APIs are properly gated with @available or #available checks, and flags any Liquid Glass usage that could crash on older OS versions.

Block 2: End-to-End Development
Full AI-driven development on a real or realistic task
~2-2.5h ▶
Choosing the Task
Option A: Real Backlog Item
  • Low-risk item <4 story points
  • No Figma dependency (or design already exists)
  • Touches code they'll work on anyway
Option B: Realistic Exercise
  • No good backlog items available
  • Items are too risky for a learning exercise
  • Want controlled scope to practice the full pipeline
The Flow
describe task → Claude implements → /ios-review-code → /ios-pre-push → /ios-pr
  • How to kick it off
    Make sure they are in the repo root with Claude Code running: cd ~/Developer/Finn/ios-app && claude Then just describe the task. Claude already has AGENTS.md context, so it knows the architecture, conventions, and available skills. Add a feature flag check for the new search filters. The toggle name is "search.filters.v2". Gate the new filter UI behind it, falling back to the existing implementation when disabled. For more complex tasks, use Plan mode first (press Shift+Tab until you see "Plan" in the status bar) to let Claude plan the approach, then switch to Normal or Auto-Accept to execute.

    After implementation, chain the repo skills: /ios-review-code # Review changes /ios-pre-push # Lint + tests + snapshots /ios-pr # Create the PR
Exercise Options (if Option B)
  • Add a feature flag to an existing screen
    Scope: ~1-2 hours. No Figma needed.

    Steps:
    1. Create a new Unleash toggle: unleash-cli create-everywhere my-feature-flag --description "Test flag" 2. In Claude Code, describe the task: Gate [existing screen element] behind the my-feature-flag Unleash toggle. Show the new version when enabled, fall back to existing when disabled. 3. Claude will modify the SwiftUI view, add the Unleash check, and write tests
    4. Run the validation chain: /ios-review-code → /ios-pre-push → /ios-pr

    This touches: Unleash CLI, SwiftUI, testing, and the full validation pipeline.
  • Port a small Android behavior to iOS
    Scope: ~1.5-2 hours.

    Find a recent simple Android PR or branch. Then invoke the porting skill: /ios-port-from-android The skill will scan the Android branch, extract API contracts and flow logic, and produce a verified iOS implementation reference. Follow with the normal review/test/PR pipeline.

    "This is one of the most impressive skills to see in action — it reads the Android code, understands the intent, and maps it to our iOS architecture automatically."
  • Write tests for an untested ViewModel
    Scope: ~1 hour. Safe, no production risk.

    Find a ViewModel with no tests: !find app-modules/Sources -name "*ViewModel.swift" | head -10 Then use /ios-write-unit-test and let Claude generate comprehensive tests. Review and discuss the test quality together.

    "This is a great way to build confidence — the changes are purely additive, there's no risk to production code, and they get to see the full AI workflow from start to finish."
Debugging & Course-Correction
  • Claude gives a wrong answer
    Step 1: Press Esc Esc to open the rewind menu and go back to before the mistake.
    Step 2: Rephrase with more context: Look at the existing pattern in app-modules/Sources/Drive/Navigation/ and follow that approach instead Step 3: If still wrong, add explicit constraints: Do NOT use UIKit. Use only SwiftUI with DriveNavigationStack. Check Documentation/NavigationPatterns.md first. "Wrong answers aren't failures — they're prompting gaps. The fix is almost always better context, not a different model."
  • MCP server fails
    /mcp # Check which servers are connected vs disconnected If a server shows disconnected:
    1. Try restarting Claude Code (exit and re-run claude)
    2. If still disconnected, re-authenticate: /mcp → select the server → follow the auth flow
    3. Most MCP failures are auth token expiration — re-authenticating usually fixes it
  • Build fails after AI changes
    First, just tell Claude about the error: The build failed. Read the error and fix it. If Claude can't fix it after one attempt, try the build-fixer approach: Use the build-fixer agent to diagnose and fix this build failure Or run pre-push to get a full diagnostic: /ios-pre-push "Build failures are normal — even human-written code breaks the build sometimes. The key is that Claude can usually fix its own mistakes if you just tell it what went wrong."
